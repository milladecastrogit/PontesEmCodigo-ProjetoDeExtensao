Sobre o Projeto

O Pontes em Código é um projeto de extensão desenvolvido para os alunos do DAX Code da Turma 2(T2 - presencial), integrando os princípios da Comunicação Não Violenta (CNV) de Marshall Rosenberg ao contexto do aprendizado de programação.

ODS 4: Educação de Qualidade

ODS 16: Paz e Justiça

Na programação colaborativa, tão importante quanto escrever código eficiente é saber comunicar-se com clareza, dar e receber feedback construtivo, e resolver conflitos de maneira empática. Este projeto oferece uma introdução prática à CNV adaptada ao universo do desenvolvimento de software.

Por que CNV para programadores?

A comunicação não violenta ajuda em situações como:

  • Code reviews construtivos (sem críticas pessoais)
  • Pair programming eficaz
  • Reuniões de equipe produtivas
  • Resolução de conflitos em projetos
  • Comunicação clara com clientes e stakeholders

Esta ferramenta foi desenvolvida para aplicação presencial com alunos do DAX Code (T2- presencial) e disponibilização via WhatsApp para quem tiver interesse. A implementação é simples e direta, focada no aprendizado prático.

Introdução à Comunicação Não Violenta

A Comunicação Não Violenta (CNV) é uma abordagem desenvolvida por Marshall Rosenberg que transforma a maneira como nos expressamos e ouvimos os outros. No contexto da programação, isso significa:

CNV no Desenvolvimento de Software

Assim como escrevemos código limpo e legível, precisamos nos comunicar com clareza e empatia. A CNV nos ajuda a:

  • Dar feedback sobre código sem criticar a pessoa
  • Expressar dificuldades técnicas sem culpar outros
  • Pedir ajuda de forma clara e específica
  • Ouvir diferentes perspectivas em discussões técnicas

A CNV é baseada em 4 componentes simples que veremos nas próximas páginas. Pense neles como "funções" para uma comunicação mais eficaz.

1. Observação (O "console.log" da Comunicação)

O que é?

Descrever fatos concretos sem julgamentos, como um console.log() mostra dados sem interpretá-los.

Exemplo no código:

  • Com julgamento: "Seu código está confuso"
  • Com observação: "Esta função tem 120 linhas e 5 níveis de indentação"

Exercício rápido: Na próxima code review, tente descrever apenas o que você vê no código, sem adicionar adjetivos como "ruim", "confuso" ou "complexo".

2. Sentimento (O "debug" emocional)

O que é?

Identificar e expressar emoções genuínas relacionadas à observação.

Exemplo no trabalho em equipe:

  • Pseudo-sentimento: "Sinto que você não se importa com o projeto"
  • Sentimento genuíno: "Estou preocupado com o prazo do projeto"

Dica para devs: Assim como identificamos bugs pelo comportamento do sistema, podemos identificar sentimentos pelas sensações no corpo (tensão, calor, frio, etc.).

3. Necessidade (Os "requisitos" humanos)

O que é?

Reconhecer quais necessidades estão por trás dos sentimentos.

Exemplo em pair programming:

  • Focando no outro: "Você precisa escrever código mais limpo"
  • Focando na necessidade: "Preciso entender o código rapidamente para contribuir"

Necessidades comuns em projetos: clareza, colaboração, respeito, aprendizado, eficiência, criatividade.

4. Pedido (A "API" da comunicação)

O que é?

Fazer solicitações claras, específicas e positivas (não exigências).

Exemplo em code review:

  • Exigência: "Você tem que refatorar isso"
  • Pedido claro: "Você poderia dividir esta função em funções menores de até 30 linhas?"

Características de um bom pedido:

  1. Ação concreta ("escrever testes" em vez de "melhorar a qualidade")
  2. Formulada positivamente ("fazer X" em vez de "não fazer Y")
  3. Permite um "não" como resposta

Resumo: O Algoritmo da CNV

1. Observo que [fato concreto]
2. Me sinto [emoção genuína]
3. Porque preciso de [necessidade]
4. Você poderia [pedido específico]?
